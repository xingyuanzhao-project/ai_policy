## A. Entry-point / script issues

- **A1. `python scripts/run_ner.py` fails with `ModuleNotFoundError: src`.** Running the script directly puts `scripts/` on `sys.path` instead of the project root, so `from src.ner...` does not resolve. Fix: either run with `python -m scripts.run_ner` from project root, or prepend `Path(__file__).resolve().parents[1]` to `sys.path` at the top of the script so click-to-run also works.
- **A2. `run_ner.py` passed no config overrides; `bootstrap.py` hardcoded `settings/` defaults.** Settings selection was buried in `src/` instead of visible at the call site. Fixed by: removing hardcoded fallbacks from `bootstrap.py`; making `config_path`, `ner_config_path`, `prompt_config_path` required `str | Path` on `pipeline_api.run_full_corpus`, `run_single_bill`, `rerun_chunk`, `rerun_group`, and `_build_runtime`; rewriting `run_ner.py` to declare all paths + `run_id` explicitly (matching the `run_skill_ner.py` pattern).

## B. Schema compatibility (Anthropic structured output)

- **B1. `anthropic/claude-sonnet-4` does not support OpenRouter structured output.** Preflight probe returned empty content. OpenRouter docs require Sonnet 4.5 or Opus 4.1+. Fix: switched `model_name` to `anthropic/claude-sonnet-4.5`.
- **B2. Anthropic rejects `maxItems`, `minItems`, `minimum`, `maximum` in JSON Schema.** Initial fix was a runtime stripper `_sanitize_schema` in `llm_client.py` applied unconditionally for `json_schema` mode. This created a second problem: `ner_prompts.json` lied about what was sent to the API. Final fix: deleted the four keywords from `ner_prompts.json` at source, removed `_UNSUPPORTED_SCHEMA_KEYWORDS` / `_sanitize_schema`, and pass `output_schema` directly. The file now reflects exactly what ships.
- **B3. `ZeroShotAnnotator._build_chunk_bounded_output_schema` / `_clamp_offset_maxima` injected `"maximum": chunk_length` into `start`/`end` integer fields at runtime.** This was invisible in the static schema but reached every API call, and Anthropic rejects `maximum` on integer types. Preflight passed, isolated tests passed, only the pipeline failed — leaving the annotator-layer injection as the only code path difference. Fix: removed both methods; annotator passes `self._output_schema` directly. Prompt already constrains offsets. One orchestration test updated to drop the old `maximum = chunk_length` assertion.

## C. Token-limit truncation

- **C1. `zero_shot_annotator.max_tokens = 768` too low.** Chunks with verbose regulatory clauses produced truncated JSON (`"text": "collect only information that is neces` → unterminated string → `JSONDecodeError`). Raised to **1024**.
- **C2. Fallback annotator `max_tokens = min(primary, 512) = 512` — *lower* than the primary it is trying to rescue.** Fallback truncated the same way. Raised the floor to **768**.
- **C3. `granularity_refiner.max_tokens = 512` too low; repeated wall-hitting.** Root cause: the refiner's `refinement_artifact` contains four `K × K` relation matrices over the group's `K` candidates; output scales as `8K² + 10K + 310`. Iterative increases (512 → 1024 → …) kept hitting new walls on denser bills. Final safe value: **16384** = Claude Sonnet 4.5's hard completion ceiling. Setting it higher risks HTTP 400 (which is *not* caught as `StageFailure` → pipeline crash). Setting below it guarantees future token-wall crashes.
- **C4. Evidence `text` spans can be 20–60+ words — the dominant token cost.** `value_evidence[].text` is verbatim legislative prose; two candidates × four fields can push a single chunk annotation past the limit. Prompt-level soft boundary added to both primary and fallback annotator prompts: *"Keep each evidence text span under 120 characters."*
- **C5. `generate()` ignored `finish_reason`.** Truncated content silently fed to `json.loads`, which failed with generic `JSONDecodeError`. Added: extract `finish_reason`, include it in the log line, raise immediately on `finish_reason == "length"`, and broaden empty-content check from `is None` to `not content`.

## D. Error propagation / exception types

- **D1. `run_corpus` bill loop only caught `CreditsExhaustedError`.** Any `StageFailure` (e.g. truncated JSON on one bill) killed the entire 1826-bill run. Fix: added `StageFailure` catch — logs the bill, appends to `failed_bills`, continues; end-of-corpus summary lists failed bills.
- **D2. `asyncio.gather` with no `return_exceptions=True` in chunk annotation.** One bad chunk killed the whole bill; `mark_stage_complete("annotation")` was never reached even though 27 of 28 chunks had persisted artifacts. Fix: wrapped `_annotate_or_resume` per chunk with `try/except StageFailure` returning `[]`; made `_annotation_stage_payload` tolerant of missing chunks via `candidate_chunk_exists` guard.
- **D3. `llm_client.generate()` raised bare `RuntimeError` on `finish_reason='length'` and on empty content.** `RuntimeError` is not a `SchemaValidationError`, so it bypassed the orchestrator's `StageFailure` wrapping (line 503/509) and propagated uncaught through `run_corpus`, crashing the pipeline. The user rejected the "change type to `SchemaValidationError` and let it skip" plan because skipping inflates NaN in results. Final shape: (a) truncation still raises and fails the bill (deterministic — retry cannot fix), (b) empty-content now retries (D5), (c) sizing is deduced from the model cap (C3) rather than reactively bumped.
- **D4. HTTP 402 (credits exhausted) was not caught and caused ungraceful crash.** `usage_summary.json` was never written because `_persist_usage_summary` lived inside the `try` body. Fix: detect `APIStatusError.status_code == 402` in `generate()`, raise new `CreditsExhaustedError(RuntimeError)`; `run_corpus` catches it, logs progress (`completed X/Y`), breaks cleanly; `_persist_usage_summary` moved into `finally` so partial totals always persist. On restart with `resume=True`, completed bills/chunks/groups are reloaded from disk.
- **D5. Empty content with `finish_reason='stop'` crashed the pipeline.** The HTTP call succeeded (200), the SDK's `max_retries` did nothing, but the model returned null/empty body. Fix: retry loop inside `generate()` driven by `self._config.max_retries`; a new `EmptyCompletionError` is raised only after all attempts return empty; truncation (`finish_reason='length'`) still raises immediately since re-sending the same prompt with the same `max_tokens` truncates again. Retry is abstracted in `generate()` so all three agents inherit it uniformly. Observed afterward on bill `2024__RI S 2540`: all 3 attempts returned **exactly 1 completion token** with `finish_reason='stop'` — Claude was deterministically refusing that particular input, so retry alone does not resolve every instance. Root cause resolved in D6.
- **D6. `native_finish_reason='refusal'` was silently discarded.** The OpenAI-normalized `finish_reason` collapses provider refusals to `"stop"`; OpenRouter preserves the upstream signal in `choices[0].model_extra["native_finish_reason"]`, and `response.model_extra["provider"]` reveals the upstream (here: `Amazon Bedrock`). The reproducer `src/ner/tests/test_refiner_empty_content_reproducer.py` confirmed temperature=0 deterministic refusal, which made the retry loop pure waste (3 identical paid calls, 3 identical refusals). Fix: (a) `llm_client.generate()` now extracts and logs both `native_finish_reason` and `provider` on every call; (b) new `RefusalError(provider, prompt_tokens, completion_tokens)` raised **without retry** when `native_finish_reason == 'refusal'`; (c) `GranularityRefiner` catches `RefusalError` and skips straight to the existing `_build_deterministic_fallback_refined_output` passthrough — the input candidate is copied through as a `RefinedQuadruplet` (no NaN, no data loss, no re-attempt of the fallback prompt which would refuse the same content); (d) `ZeroShotAnnotator` treats primary-refusal → fallback-prompt, fallback-refusal → re-raise as `SchemaValidationError` so the per-chunk handler drops that chunk only; (e) `EvalAssembler` treats refusal → `_build_fallback_groups` (deterministic one-group-per-candidate grouping). Every refusal produces two WARNING log lines: a low-level one from `llm_client` (provider, tokens, finish reasons) and a high-level one from the agent with bill/chunk/group/candidate context and `action=deterministic_passthrough`. Verified end-to-end: 43/43 offline tests + live reproducer against `2024__RI S 2540` group `7945457209652008633` now makes 1 API call instead of 3, passes through deterministically, logs the refusal with full context.

## E. Logging / observability

- **E1. `run.log` was empty after a ~78-minute run.** `_attach_run_log_handler` added a `FileHandler` at level INFO to the root logger, but the root logger's own default level is `WARNING` — every `logger.info(...)` was filtered before reaching the handler, so the file was created but never written. Fix inside `bootstrap.py`: after attaching the handler, `if root.level > logging.INFO: root.setLevel(logging.INFO)`. Script-level `logging.basicConfig` is *not* required for the file log (only for console streaming).
- **E2. No per-request / per-stage observability existed initially.** Added INFO logs across four files: `llm_client.py` (connection params, preflight, per-call `elapsed_ms` + `prompt_tokens` + `completion_tokens` + `finish_reason` + `native_finish_reason` + `provider` + `attempt=N/M` + `cost=$...`), `bootstrap.py` (resolved base_url / model / mode / key source, plus per-agent `temperature` + `max_tokens`, runtime `concurrency` + `chunk_size` + `overlap` + `max_bill_text_chars` + `storage_dir` — one glance at the run.log header now reproduces the full runtime configuration without re-running), `orchestrator.py` (per-bill per-stage timing: annotation, grouping, refinement, total), `inference_loop.py` (bill start/skip, corpus start/finish).
- **E3. OpenRouter's `response.usage.cost` / `reasoning_tokens` / `cached_tokens` were never extracted.** Only prompt/completion tokens were logged; dollar cost was not recorded. Fix: added `UsageStats` thread-safe accumulator in `llm_client.py` that pulls `cost`, `reasoning_tokens`, `cached_tokens`, `total_tokens` from every response; `usage_summary.json` now writes cumulative USD cost + token breakdown to the run directory.

## F. Resume / usage-summary persistence

- **F1. `_persist_usage_summary(context)` was inside the `try` block of `run_full_corpus` / `run_single_bill`.** On crash it never ran, so partial session cost was lost. Moved into `finally` in both entry points; the `FileHandler` is detached in the same `finally` via `context.detach_run_log()`.
- **F2. On resume, the fresh session would overwrite the previous `usage_summary.json`.** Fix in `_persist_usage_summary`: if the file already exists, load the previous values and add them to the current session's totals before writing. Verified: after two identical 5-bill runs, merged file shows exactly `2 × single-run` totals.

## G. Multi-year corpus ingestion

- **G1. Year-unqualified `bill_id` collides across years.** The same source `bill_id` appears in 2023, 2024, and 2025 (72 collisions across the merged corpus). Fix in `corpus_store._row_to_bill_record`: compose `qualified_bill_id = f"{year}__{raw_bill_id}"`; added `year` and `source_bill_id` fields to `BillRecord` (both default `""` for backward compatibility). After fix: 1879 rows → 53 empty-text filtered → 1826 processed; 3 remaining collisions are same-year duplicates with identical text (e.g. `TX H 1` ×4 in 2023) harmlessly deduplicated by the `dict`.
- **G2. Empty-text rows (53 of 1879) were passed into the pipeline.** Added filtering + logging in `corpus_store.load()`.
- **G3. Wrong input file risk.** `data/ncsl/ncsl_bills_2023.jsonl` and `ncsl_bills_2025.jsonl` are raw HTML scrapes with only `bill_id`, `html`, `year` — no `state`, no `text`. `validate_bill_record` requires non-empty `state`, so pointing `input_path` at these raises `SchemaValidationError`. Only `us_ai_legislation_ncsl_text*.jsonl` files are valid pipeline input. Extra columns (`text_len`, `word_count`, `extraction_status`, `n_chars`, `n_words`, `year`) in the text files are silently ignored by `row.get(...)` — no code change needed to consume them.
- **G4. Final output JSON carried no bill metadata.** Downstream evaluation could not join outputs back to year + state + `source_bill_id`. Fix: added `_bill_metadata` registry and `register_bill_metadata()` to `FinalOutputStore`; `save()` now wraps the refined list in `{bill_id, source_bill_id, year, state, quadruplets}`; `load()` / `load_all()` handle both the new wrapped format and the legacy bare-list format.
- **G5. `InferenceLoop.__init__` needed a `final_output_store` parameter** so it could call `register_bill_metadata()` before each orchestration. `entry_points.py` now passes `context.final_output_store`. `test_inference_loop.py` updated to pass a `Mock()` and assert metadata registration.
- **G6. `scripts/test_openrouter_nbills.py` display loop used raw `bill_id`** but pipeline output keys are year-qualified; fixed to construct the year-qualified id for lookup + wider column header.

## J. Research-level findings (critical)

- **J1. Commercial LLM providers refuse requests on politically-, chemically-, or technologically-sensitive content.** Bill `2024__RI S 2540` — the *Rhode Island Clean Air Preservation Act*, anti-geoengineering / chemtrails legislation — was deterministically refused by `anthropic/claude-sonnet-4.5` served through Amazon Bedrock: every attempt returned `completion_tokens=1`, `finish_reason='stop'`, `native_finish_reason='refusal'`, reproduced 3/3 at temperature 0, and confirmed in the Claude GUI (same safety filter fires interactively). The bill is unambiguously public law, not user-generated sensitive content, yet the provider's safety layer refused to produce structured output for it. **This is a standing concern for using commercial LLMs in social-science research on legislation**: any corpus touching environmental regulation, chemical/biological agents, surveillance, national security, elections, health policy, or contested political topics will hit provider-side refusals that are (a) deterministic (retry cannot bypass them), (b) invisible if `native_finish_reason` is not surfaced, and (c) silently skewing the analyzable sample toward "safe" bills. The technical mitigation (D6) ensures the pipeline does not *crash* on refusal and does not *drop* the input, but the refused content still gets no LLM analysis — it is passed through as the deterministic candidate. Downstream evaluation must treat refused bills as a systematic missing-data category, not as normal output.

- **J2. The NER classifications may be overly positive: the annotator extracts content that is not actually about AI.** Bill `2024__RI S 2540` reached the corpus because NCSL's keyword filter matched the word "Artificial intelligence" in a single definitions section; the **body** of the bill is about stratospheric aerosol injection, radio-frequency radiation, xenobiotic agents, and state-police enforcement against geoengineering operators. The zero-shot annotator then extracted the candidate `entity="facility or infrastructure deploying or releasing the specific agent or agents"`, `type="infrastructure"`, `attribute="restriction"`, `value="cease operations"` — where **"agent" means *chemical/biological agent*, not *AI agent*** (the bill's own definitions 7/10/15/17/19 make this explicit). The annotator has no disambiguation step: it matches the surface string "agent" and produces a quadruplet that looks like AI policy but is in fact chem/environmental policy. Structural tells in the extracted quadruplet — disjunctive generic entity phrase (`facility or infrastructure`), back-reference to unstated context (`the specific agent or agents`), identical entity- and type-evidence spans (start=16871, end=16949 for both), attribute/value evidence from a disjoint clause (chars 17165–17225) stitched onto the entity (chars 16871–16949) — are not caught downstream. Net effect: **the fixed pipeline will mark as "AI-policy quadruplets" material that a human reader would immediately reject.** This is a precision problem (false positives), not a recall problem, and it is invisible in the current output unless the evidence spans and source bill are manually inspected.

- **J3. The fixed 3-stage pipeline (`src/ner/`) is worse than the skill-driven agentic pipeline (`src/skill_ner/`) along three axes, demonstrated by this debugging saga.** 
  1. **Exception handling is harder.** The fixed pipeline has a fragile exception chain: `RuntimeError` (from `finish_reason='length'` or refusal) bypasses `StageFailure`; `asyncio.gather` without `return_exceptions=True` kills a bill on one bad chunk; `CreditsExhaustedError` and `RefusalError` each needed bespoke plumbing across `llm_client` → agent → orchestrator → `inference_loop`. Every error class required (a) a new exception type, (b) a catch at the right layer, (c) a routing decision (retry / fallback prompt / deterministic passthrough / skip-chunk / skip-bill / break-corpus), (d) a log line, (e) tests. Issues D1–D6 and F1–F2 all exist because the 3-stage pipeline forces this cross-layer bookkeeping. A skill-driven agentic loop collapses the choice of "what to do on failure" into the agent's own step logic (retry / re-plan / abandon-subtask) and does not need a separate exception type per failure mode.
  2. **Debugging cost is higher.** Every fix in this log revealed the next wall. Schema keywords (B2) → runtime schema injection (B3) → `max_tokens` too small (C1–C3) → `finish_reason` ignored (C5) → `RuntimeError` vs. `StageFailure` (D3) → HTTP 402 not caught (D4) → empty content (D5) → refusal signal discarded (D6). Each fix was narrow, deterministic, and necessary — but there was no single diagnostic that predicted the next wall, because the staged pipeline has no reflective awareness of its own state. An agentic loop can inspect its own failure, re-plan, and surface the actionable signal in one place rather than across five files.
  3. **Keyword over-matching is a structural risk.** The annotator's zero-shot extraction has no way to recognise that "agent" in `2024__RI S 2540` means *chemical agent*. The fixed pipeline trusts the output of each stage and has no step where "does this bill actually belong in an AI-policy corpus?" is asked. A skill-driven agentic annotator can condition extraction on a preceding relevance check (e.g., "read the bill; is this actually about AI? if not, abstain"), which is exactly the kind of compound skill the agentic framework in `src/skill_ner/` is built for. The fixed pipeline cannot add that check without becoming another stage with another exception class.
  These three axes argue for preferring `src/skill_ner/` for future NER work on this corpus, and for treating `src/ner/` outputs as a baseline to be diffed against the agentic results rather than as ground truth.
